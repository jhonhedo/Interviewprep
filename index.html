Core Java

Check off each subtopic as you complete it. Expand cards to see questions and resources.

A OOP & Java Fundamentals ~4 hrs ▶
90% of mid-level interviews start with OOP scenarios. Know how to apply, not just define each principle.
Study checklist
  • Polymorphism: runtime (method overriding) vs compile-time (overloading) — with real examples must
  • Method hiding (static) vs method overriding (instance) — why they differ
  • Abstract class vs Interface — when each, Java 8+ default & static methods must
  • Immutability: String pool, how to design a fully immutable class with mutable fields must
  • equals() and hashCode() contract — what breaks if you violate it (broken HashMap behavior)
  • final keyword: final class, final method, final variable, blank final
  • static: class-level members, static initializer blocks, static import
  • Covariant return types and constructor chaining (this(), super())
  • Exception hierarchy: checked vs unchecked, custom exceptions, try-with-resources
Interview questions
Why is String immutable in Java? What's the benefit for security and performance?
Can you override a static method? What happens if you try? What is method hiding?
Design an immutable class that contains a List field.
If I use a mutable object as a HashMap key and then mutate it — what happens?
What is the difference between an abstract class and an interface in Java 8+?
B Collections & Generics ~3 hrs ▶
HashMap internals are the #1 asked topic across all companies. Know it cold — load factor, rehashing, treeification at threshold 8.
Study checklist
  • HashMap internals: hash(), index calculation, collision resolution, Java 8 treeification at 8 nodes must
  • Load factor (0.75) and rehashing — when it triggers, what it costs
  • LinkedHashMap (insertion order), TreeMap (sorted), EnumMap — when each
  • ConcurrentHashMap: segment-level locking (Java 7) → CAS + synchronized bucket (Java 8) must
  • ArrayList vs LinkedList — memory layout, O(n) access vs O(n) insert, practical choice
  • Fail-fast vs fail-safe iterators — ConcurrentModificationException cause
  • Comparable vs Comparator — chaining comparators in Java 8
  • Generics: bounded wildcards (? extends T, ? super T), type erasure, reifiable types
  • PriorityQueue: min-heap by default, custom Comparator for max-heap
Interview questions
Explain step-by-step what happens when you call put(key, value) on a HashMap.
When would you use LinkedHashMap over HashMap?
What is the difference between fail-fast and fail-safe iterators? Give an example of each.
Explain type erasure. Why can't you do new T() inside a generic class?
C Multithreading & Concurrency ~5 hrs ▶
Asked heavily at product companies. Focus on practical patterns and code — not just theory.
Study checklist
  • Thread lifecycle: NEW → RUNNABLE → BLOCKED → WAITING → TIMED_WAITING → TERMINATED
  • Thread vs Runnable vs Callable vs Future — when each is appropriate
  • ExecutorService: fixed, cached, single, scheduled thread pools — use cases must
  • synchronized: method-level vs block-level, intrinsic lock, reentrant behavior
  • volatile: visibility guarantee only (NOT atomicity) — memory model happens-before must
  • ReentrantLock, ReadWriteLock, StampedLock — advantages over synchronized
  • Atomic classes: AtomicInteger, AtomicReference — CAS (compare-and-swap) operation
  • Deadlock: cause, 4 conditions (Coffman), detection, prevention, lock ordering must
  • wait() / notify() / notifyAll() — why they must be inside synchronized block
  • CompletableFuture: thenApply, thenCompose, thenCombine, allOf, anyOf, exceptionally
  • ThreadLocal — use case (per-thread request context), memory leak risk in thread pools
  • Producer-consumer pattern using BlockingQueue (ArrayBlockingQueue, LinkedBlockingQueue)
Interview questions
Implement a thread-safe Singleton using double-checked locking. Why do you need volatile?
What is a race condition? Show a code example and fix it with synchronized and then with AtomicInteger.
Difference between wait() and sleep()? Why is wait() always inside synchronized?
Implement producer-consumer without BlockingQueue using wait/notifyAll.
What is a thread pool? Why is it dangerous to use Executors.newCachedThreadPool() in production?
D Java 8–17 Features ~4 hrs ▶
Streams are non-negotiable. If you can't write stream pipelines fluently, you look junior regardless of your experience.
Study checklist
  • Stream pipeline: source → intermediate ops → terminal op. Lazy evaluation. must
  • map vs flatMap — flattening nested structures with flatMap
  • Collectors: groupingBy, partitioningBy, toMap, joining, counting, summingInt must
  • Optional: orElse vs orElseGet vs orElseThrow, map, flatMap, ifPresent — anti-patterns
  • Functional interfaces: Predicate, Function, Consumer, Supplier, BiFunction, UnaryOperator
  • Method references: static (Class::method), instance (obj::method), constructor (Class::new)
  • var (Java 10): local type inference — when to use and when it hurts readability
  • Record classes (Java 16+): compact constructor, custom methods, immutability by default
  • Sealed classes (Java 17): restrict subclassing, use with pattern matching switch
  • Text blocks (Java 15), switch expressions (Java 14+), pattern matching instanceof
Interview questions
Group a list of employees by department and find the maximum salary in each group using streams.
What is the difference between map() and flatMap()? Give a concrete example.
When would you NOT use Optional? What are its pitfalls in method parameters?
What is the difference between orElse() and orElseGet()? When does it matter?

JVM Internals & Memory Management

Often underestimated — but asked at senior-leaning mid-level roles and in performance debugging scenarios.

E JVM Architecture & Class Loading ~3 hrs ▶
Understanding JVM architecture is the difference between a developer who "writes Java" and one who "understands the platform." Expect this in senior 3+ YoE interviews.
Study checklist
  • JVM components: Class Loader, Runtime Data Areas, Execution Engine, JNI must
  • Class loading phases: Loading → Linking (Verify, Prepare, Resolve) → Initialization
  • Class loaders: Bootstrap → Extension → Application — parent delegation model must
  • Custom ClassLoader — when would you write one? (hot reload, isolation)
  • JIT compilation: interpreted → compiled hotspot methods, tiered compilation levels 0–4
  • Bytecode basics: .class file structure, javap -c output, how JVM executes bytecode
Interview questions
Explain the parent delegation model in class loading. Why does it exist?
What happens step by step when you run java MyClass?
Can two classes with the same fully qualified name coexist in a JVM? How?
F Memory Areas: Heap, Stack, Metaspace ~3 hrs ▶
Memory-related bugs (OOM, memory leaks) come up in real production scenarios. Knowing the memory model proves operational maturity.
JVM Memory areas checklist
  • Heap: where all objects live, shared across threads, GC-managed must
  • Heap generations: Young Gen (Eden + S0 + S1), Old Gen (Tenured), why this split exists
  • Stack: per-thread, holds frames (local vars + operand stack + reference to constant pool)
  • StackOverflowError vs OutOfMemoryError — causes and how to diagnose must
  • Metaspace (Java 8+): replaced PermGen, holds class metadata, grows dynamically
  • String Pool: interning, == vs equals() for strings, where pool lives (heap since Java 7)
  • PC Register and Native Method Stack — purpose and stack trace role
  • Common JVM flags: -Xms, -Xmx, -Xss, -XX:MetaspaceSize, -XX:+HeapDumpOnOutOfMemoryError
Interview questions
What is the difference between heap and stack in the JVM?
Why was PermGen replaced with Metaspace in Java 8?
Your application gets OutOfMemoryError: GC overhead limit exceeded. What do you do?
Where do static variables live in the JVM?
G Garbage Collection ~4 hrs ▶
GC tuning questions signal that you've worked in production. Even basic GC knowledge sets you apart from most 3 YoE candidates.
Study checklist
  • GC roots: what keeps an object alive (local vars, static fields, JNI refs, thread stacks) must
  • Minor GC (Young Gen): Eden allocation, survivor copy, promotion to Old Gen
  • Major/Full GC: Old Gen collection, Stop-The-World pause, when it triggers
  • GC algorithms: Serial, Parallel, CMS (deprecated), G1 (default Java 9+), ZGC/Shenandoah (low-latency) must
  • G1GC: region-based heap, mixed GC, pause time targets (-XX:MaxGCPauseMillis)
  • Memory leaks in Java: static collections, ThreadLocal misuse, listener not removed, inner classes
  • Reference types: Strong, Soft (cache), Weak (WeakHashMap), Phantom — when each is used
  • Heap dumps: generate with jmap or -XX:+HeapDumpOnOutOfMemoryError, analyze with VisualVM/Eclipse MAT
  • GC logging: -Xlog:gc* (Java 9+), how to read GC pause times, throughput vs latency trade-off
  • finalize() — why it's deprecated (Java 9+), Cleaner API as replacement
Interview questions
Explain how G1GC works. Why is it the default since Java 9?
What is a memory leak in Java? Give 3 real examples and how to fix them.
Difference between a Soft reference and a Weak reference. When would you use each?
How would you diagnose a production Java service that is using too much memory?
What is Stop-The-World? How do modern GCs minimize it?
H Performance & Profiling Tools ~2 hrs ▶
Study checklist
  • jstack: thread dump, deadlock detection, thread state analysis
  • jmap: heap dump, object histogram (-histo), live objects
  • jstat: GC statistics, class loading, JIT compilation stats
  • VisualVM / JConsole: real-time monitoring, heap graph, thread view
  • JMH (Java Microbenchmark Harness): correct way to benchmark Java code — avoid JVM warmup traps
  • Common performance pitfalls: String concatenation in loops, unnecessary boxing, stream vs for-loop

Spring Boot & Microservices

Practical and real-world scenarios — internals over config knowledge.

I Spring Core & Boot Internals ~4 hrs ▶
Interviewers test "what happens under the hood," not just @Autowired usage. Know bean lifecycle cold.
Study checklist
  • IOC container: BeanFactory vs ApplicationContext, eager vs lazy initialization must
  • Bean lifecycle: Constructor → @PostConstruct → business use → @PreDestroy
  • Bean scopes: Singleton (default), Prototype, Request, Session, Application must
  • @Component vs @Service vs @Repository vs @Controller — functional differences
  • @Component vs @Bean — when to use each
  • @Transactional: propagation (REQUIRED, REQUIRES_NEW, NESTED), isolation, rollbackFor must
  • @Transactional self-invocation problem — why calling from within same class breaks it
  • AOP: @Aspect, @Before, @After, @Around, @Pointcut — practical logging/timing use case
  • Circular dependency: causes, detection, resolution (@Lazy, setter injection, refactoring)
  • Auto-configuration: @SpringBootApplication = @Configuration + @ComponentScan + @EnableAutoConfiguration
  • Spring profiles: @Profile, application-{profile}.yml, activating via env variable
Interview questions
What is the difference between @Component and @Bean?
Explain @Transactional propagation: REQUIRED vs REQUIRES_NEW — give a scenario for each.
How would you handle a circular dependency in Spring?
What happens if you call a @Transactional method from within the same class?
J REST APIs, Security & JPA ~5 hrs ▶
Almost every real project uses JPA + REST + JWT. This is the bread-and-butter of the interview and must be flawless.
Study checklist
  • REST best practices: HTTP verbs, idempotency, correct status codes (200/201/204/400/401/403/404/409/500)
  • @ControllerAdvice + @ExceptionHandler — global exception handling, custom error response body
  • Spring Security filter chain: OncePerRequestFilter, JwtAuthFilter placement must
  • JWT: header.payload.signature, signing algorithms (HS256, RS256), token validation steps
  • JPA N+1 problem: diagnosis with Hibernate logging, fix with JOIN FETCH or @EntityGraph must
  • Lazy vs Eager loading — LazyInitializationException cause and fixes
  • Pagination with Pageable, sorting with Sort, projection interfaces for DTO queries
  • @Query with JPQL vs native SQL, named parameters, @Modifying for updates/deletes
  • Redis caching: @Cacheable, @CacheEvict, @CachePut, cache key strategies
Interview questions
Explain the N+1 query problem. How do you diagnose it and fix it in Spring Data JPA?
How does JWT authentication work in a Spring Boot filter chain step-by-step?
What is the difference between @OneToMany with EAGER vs LAZY loading?
How would you design a global exception handler for a REST API?
K Microservices Patterns ~4 hrs ▶
Know patterns by name and code — not just what they are but when and why you'd use them.
Study checklist
  • Service discovery: Eureka — registration, heartbeat, client-side load balancing must
  • API Gateway: Spring Cloud Gateway — routing, filters, rate limiting, auth
  • Feign client: declarative HTTP, @FeignClient, fallback with Resilience4j
  • Circuit breaker (Resilience4j): CLOSED → OPEN → HALF_OPEN states, configuration must
  • SAGA pattern: choreography (events) vs orchestration (central coordinator) — trade-offs
  • Kafka basics: topic, partition, consumer group, offset, at-least-once vs exactly-once delivery
  • Distributed tracing: Micrometer + Zipkin, trace ID propagation across services
  • Outbox pattern: reliable event publishing without dual-write problem
Interview questions
How do you handle partial failure when calling 3 downstream services synchronously?
What is a circuit breaker? Walk through all state transitions with trigger conditions.
How do you maintain data consistency across microservices without distributed transactions?
What is the SAGA pattern? Compare choreography vs orchestration — when to choose each.

DSA Patterns

Pattern-first approach. Master the pattern, apply to multiple problems. Aim for 80–100 problems total.

L Arrays & Two Pointers ~6 problems ▶
Two-pointer is the foundation of 30% of array problems. Learn to recognize the pattern instantly.
Concepts checklist
  • Two-pointer on sorted array: when to move left, when to move right pattern
  • Fast/slow pointer (Floyd's cycle detection): linked list and array cycles
  • Prefix sum: subarray sum queries in O(1) after O(n) precompute must
  • Kadane's algorithm: maximum subarray sum in O(n)
  • Dutch National Flag (3-way partition): sort 0s, 1s, 2s in one pass
Practice problems
  • Two Sum — LC #1 easy
  • 3Sum — LC #15 medium
  • Container With Most Water — LC #11 medium
  • Trapping Rain Water — LC #42 hard
  • Maximum Subarray (Kadane's) — LC #53 medium
  • Subarray Sum Equals K (prefix sum + hashmap) — LC #560 medium
M Sliding Window ~5 problems ▶
Sliding window converts O(n²) brute-force substring/subarray problems to O(n). Recognize it by the "contiguous" constraint.
Concepts checklist
  • Fixed-size window: maintain sum/count as window slides pattern
  • Variable-size window: expand right, shrink left when constraint violated must
  • HashMap frequency tracking inside window for character-based problems
Practice problems
  • Longest Substring Without Repeating Characters — LC #3 medium
  • Minimum Window Substring — LC #76 hard
  • Permutation in String — LC #567 medium
  • Longest Repeating Character Replacement — LC #424 medium
  • Maximum Sum Subarray of Size K — custom easy
N HashMap & Hashing Patterns ~5 problems ▶
Concepts checklist
  • Frequency count: count occurrences, find majority element, top K frequent pattern
  • Grouping by key: anagram grouping (sort as key), group by remainder
  • Complement pattern: store seen values, check complement exists (Two Sum base)
  • Index map: store first occurrence index to compute max length subarray/substring
Practice problems
  • Group Anagrams — LC #49 medium
  • Top K Frequent Elements — LC #347 medium
  • Longest Consecutive Sequence — LC #128 medium
  • LRU Cache (design with LinkedHashMap or custom) — LC #146 medium
O Binary Search ~6 problems ▶
Binary search on answer space (not just sorted array) is a mid-level must-know. Recognize it by monotonic feasibility functions.
Concepts checklist
  • Standard binary search template: lo, hi, mid = lo + (hi - lo) / 2, avoid overflow must
  • Left/right boundary search: first occurrence, last occurrence (separate templates)
  • Binary search on rotated sorted array: determine which half is sorted, then narrow must
  • Binary search on answer space: feasibility function + monotonicity = binary search on result pattern
  • Peak element: binary search on unsorted array using slope direction
Practice problems
  • Binary Search — LC #704 easy
  • Search in Rotated Sorted Array — LC #33 medium
  • Find Peak Element — LC #162 medium
  • Koko Eating Bananas (answer space) — LC #875 medium
  • Minimum Days to Make m Bouquets (answer space) — LC #1482 medium
  • Median of Two Sorted Arrays — LC #4 hard
P Stack & Monotonic Stack ~5 problems ▶
Concepts checklist
  • Balanced parentheses pattern: push open, pop and match on close pattern
  • Monotonic decreasing stack: next greater element — pop when current > top must
  • Monotonic increasing stack: previous smaller element — pop when current < top
  • Largest rectangle in histogram: extend bar using stack of indices
Practice problems
  • Valid Parentheses — LC #20 easy
  • Daily Temperatures — LC #739 medium
  • Next Greater Element I — LC #496 easy
  • Largest Rectangle in Histogram — LC #84 hard
  • Car Fleet — LC #853 medium
Q Graphs: BFS & DFS ~7 problems ▶
Concepts checklist
  • Graph representations: adjacency list vs adjacency matrix — space/time trade-off must
  • BFS template: queue + visited set, level-by-level traversal pattern
  • DFS template: recursion (call stack) vs iterative (explicit stack), visited tracking
  • BFS for shortest path in unweighted graph (layers = distance) must
  • Topological sort: Kahn's (BFS with in-degree) and DFS-based
  • Union-Find (Disjoint Set): find with path compression, union by rank pattern
  • Cycle detection: directed graph (DFS with state: WHITE/GRAY/BLACK), undirected (parent tracking)
Practice problems
  • Number of Islands — LC #200 medium
  • Clone Graph — LC #133 medium

  • Pacific Atlantic Water Flow — LC #417 medium

R Trees & Heaps ~8 problems ▶
Concepts checklist
  • DFS traversals: inorder (sorted BST), preorder (copy tree), postorder (delete tree) must
  • BFS level-order traversal: queue-based, collect by level pattern
  • BST operations: insert, delete, search — all O(h), O(log n) balanced
  • Lowest Common Ancestor (LCA): recursive approach for BST and general binary tree
  • Tree diameter, max path sum: DFS returning max depth, tracking global max
  • Heap / PriorityQueue: min-heap by default, invert comparator for max-heap must
  • Top K pattern: use min-heap of size K to maintain K largest — O(n log k)
  • Two heaps pattern: max-heap left half + min-heap right half → median in O(log n) pattern
Practice problems
  • Binary Tree Level Order Traversal — LC #102 medium
  • Maximum Depth of Binary Tree — LC #104 easy
  • Lowest Common Ancestor of BST — LC #235 medium

  • Merge K Sorted Lists — LC #23 hard
  • Find Median from Data Stream — LC #295 hard
S Dynamic Programming (Essential) ~7 problems ▶
At 3 YoE, focus on 1D and 2D DP — not complex DP. Be able to identify subproblem, formulate recurrence, and optimize space.
Concepts checklist
  • Memoization (top-down): recursion + cache — use HashMap or array must
  • Tabulation (bottom-up): fill DP array iteratively, identify base cases
  • Space optimization: when only previous row/two cells needed, use O(1) or O(n) space
  • 0/1 Knapsack pattern: include or exclude current item, 2D table → 1D optimization pattern
  • Unbounded knapsack: Coin Change — item can be used multiple times
  • LCS / LIS patterns: 2D table for LCS, patience sorting for LIS in O(n log n)
Practice problems

  • Coin Change — LC #322 medium
  • Longest Common Subsequence — LC #1143 medium
  • Longest Increasing Subsequence — LC #300 medium

  • Word Break — LC #139 medium

SQL & Database Optimization

Real interview questions and performance tuning scenarios.

T SQL Queries & Window Functions ~4 hrs ▶
Window functions separate mid-level from junior in SQL rounds. If you can write ROW_NUMBER + PARTITION BY fluently, you've already won half the battle.
Study checklist
  • All JOIN types: INNER, LEFT, RIGHT, FULL OUTER, CROSS, SELF — with practical use cases must
  • CTE (WITH clause) vs subquery — readability and when CTE is materialized
  • ROW_NUMBER, RANK, DENSE_RANK — difference with tie handling example must
  • LAG, LEAD: access previous/next row — running difference, period-over-period comparison
  • SUM / AVG OVER (PARTITION BY ... ORDER BY ...): running total, moving average
  • GROUP BY vs PARTITION BY — one collapses rows, other doesn't
  • HAVING vs WHERE — filter on aggregate vs filter on row
  • NULL handling: COALESCE, NULLIF, IS NULL vs = NULL (always false)
Interview questions
Find the second-highest salary in each department (without LIMIT).
Find employees who earn more than their manager (self-join).
Get running total of sales by date using a window function.
Find and delete duplicate rows keeping only the one with the lowest id.
U Indexes, Query Optimization & Transactions ~4 hrs ▶
Study checklist
  • Index types: B-tree (default), hash, full-text, composite, covering index must
  • Composite index leftmost prefix rule: (A, B, C) — queries on A, A+B, A+B+C use it; B alone does not
  • When NOT to index: low cardinality columns, high write / low read tables, small tables
  • EXPLAIN / EXPLAIN ANALYZE: read query plan, spot full table scans, index range scans must
  • ACID: Atomicity, Consistency, Isolation, Durability — real-world examples for each
  • Isolation levels: READ UNCOMMITTED (dirty read), READ COMMITTED, REPEATABLE READ (phantom read), SERIALIZABLE must
  • Deadlock: cause, detection (SHOW ENGINE INNODB STATUS), resolution (retry + lock ordering)
  • Normalization (1NF–3NF) vs denormalization — when to denormalize for read performance
  • SQL vs NoSQL: when to choose — relational integrity vs horizontal scale vs schema flexibility
Interview questions
A query on a 10M row table is slow. Walk through your step-by-step debug approach.
You have an index on (A, B, C). Does a query filtering on B and C use the index?
What is a phantom read? Which isolation level prevents it and what is the trade-off?
When would you prefer a NoSQL database over a relational database? Give 3 scenarios.

System Design

Beginner to intermediate backend design. Focus on structured thinking and trade-offs.

V Core Building Blocks ~5 hrs ▶
At 3 YoE, demonstrate structured thinking and vocabulary. You don't need perfect answers — you need to show you reason through trade-offs.
Study checklist
  • Load balancing: round-robin, least connections, IP hash, consistent hashing must
  • Caching: cache-aside, write-through, write-behind, TTL, eviction policies (LRU, LFU)
  • Redis: data structures (string, hash, list, set, sorted set), pub/sub, distributed lock
  • Message queues: Kafka vs RabbitMQ — ordering, durability, consumer groups, use cases
  • Database sharding: horizontal partitioning, range vs hash sharding, hotspot problem
  • CAP theorem: Consistency, Availability, Partition tolerance — real system examples for each pair
  • Consistent hashing: minimize redistribution when nodes added/removed, virtual nodes
  • SQL vs NoSQL: Cassandra (time-series/wide-column), MongoDB (documents), Redis (cache), Elasticsearch (search)
  • Design interview framework: Clarify → Estimate → High-level diagram → Deep dive → Trade-offs
Must-practice designs (30 min each, out loud)
  • URL shortener — hashing, redirect, analytics, DB choice, caching
  • Rate limiter — token bucket vs sliding window log vs fixed window counter
  • Notification system — email/SMS/push, fan-out, delivery guarantees
  • Payment processing — idempotency, saga pattern, retry strategy
  • Chat system — WebSocket, message storage, read receipts, presence
  • E-commerce product search — Elasticsearch, filters, ranking, caching

Low-Level Design

OOP, SOLID, and design patterns — with Java code examples.

W SOLID Principles with Java Code ~3 hrs ▶
Study checklist
  • SRP: one reason to change — UserService doing email sending = SRP violation; split it must
  • OCP: add discount types without modifying DiscountService — use Strategy/polymorphism
  • LSP: Square extending Rectangle — area calculation bug. Know the violation and fix must
  • ISP: large IAnimal interface → split into IFlyable, ISwimmable, IRunnable
  • DIP: depend on UserRepository interface, not UserRepositoryImpl — enables mocking
Design patterns to code (not just describe)
  • Singleton: enum-based (thread-safe, serialization-safe) + double-checked locking with volatile must
  • Factory + Abstract Factory: decouple creation — DatabaseFactory creating MySQL/Postgres connections
  • Builder: User.builder().name("x").email("y").build() — used everywhere in Java libraries must
  • Strategy: replace if-else payment type handling with PaymentStrategy interface
  • Observer: event system — listeners, EventBus, real use in Spring (ApplicationEventPublisher)
  • Decorator: wrapping InputStream, adding functionality without subclassing
  • Template Method: OrderProcessor defines flow, subclasses override specific steps
LLD practice problems (implement with classes, not just explain)
  • Parking lot: levels, vehicle types, ticket system, fee calculator must
  • Library management system: book, member, issue/return, search
  • Food delivery system: restaurant, rider, order, status FSM
  • ATM machine: card states, PIN validation, balance, withdraw
  • Chess game: board, pieces, move validation, turn management

Projects & Resume

1 strong project beats 5 CRUD apps. Add depth: async, caching, auth, monitoring.

X Project Checklist — Must-Have Features Weeks 7–8 ▶
Pick one project idea
  • E-commerce backend: product catalog, cart, order, inventory, payment simulation
  • Job portal microservices: 3+ services (user, job, notification) + Kafka + Docker Compose
  • URL shortener with analytics: custom alias, Redis caching, rate limiting, click tracking
Must have in every project
  • JWT authentication + role-based authorization (@PreAuthorize)
  • Global exception handler with proper HTTP status codes and error body
  • Pagination, sorting, filtering via Spring Data Pageable
  • Redis caching on at least one high-read endpoint (@Cacheable)
  • Unit tests with JUnit 5 + Mockito — minimum 60% coverage, test edge cases
  • README with architecture diagram, API endpoints, setup instructions
  • Deployed on Railway/Render or fully Dockerized with docker-compose.yml
  • API documentation with Swagger/OpenAPI (@Operation annotations)
Resume rules
  • 1 page only — no photo, no objective statement
  • Each bullet: Action verb + What + Result. Bad: "Worked on APIs." Good: "Built 12 REST APIs reducing avg response time by 35% via Redis caching."
  • Skills: Java, Spring Boot, Hibernate, MySQL, Redis, Kafka, Docker, Git, JUnit — no "MS Word"
  • ATS-friendly: no tables, columns, graphics, headers/footers — plain text with proper headings
  • Tailor skills section to each job description — match their exact keywords
Behavioral prep — STAR format
  • Tell me about yourself — 90-second pitch: current role → impact → why looking
  • Biggest challenge — real technical debugging story, focus on your decision process
  • Conflict with colleague — focus on resolution and what you learned, not blame
  • Greatest achievement — tie to business impact (latency, revenue, scale), not just code quality
  • Why this company — research 2–3 specific things: tech stack, product, engineering blog
  • Salary negotiation: never give first number, know your market rate, get competing offers

  • Record yourself — watch back for filler words and pace
  • At least 1 full mock per week from Week 4 onward
  • Use friends / colleagues for informal mock rounds

Technical round framework

  • DSA: think out loud, brute force first, then optimize
  • System design: Clarify → Estimate → Diagram → Deep dive → Trade-offs
  • LLD: draw class diagram first, then code one class at a time
  • Always ask clarifying questions — signals seniority

Negotiation tips

  • Never give a number first — ask for their budget range
  • Get 2+ competing offers for real leverage
  • Negotiate base, bonus, joining bonus, ESOPs separately
  • Know your BATNA before every offer call
